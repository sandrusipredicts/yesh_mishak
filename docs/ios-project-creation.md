# iOS Project Creation (Capacitor)

## Summary

This document records the creation of the first Capacitor iOS project for the app, connecting the existing React/Vite frontend to a native Xcode project. `@capacitor/ios` was installed, `frontend/ios/` was generated via `npx cap add ios`, and synced via `npx cap sync ios`. Generation and sync both completed with no errors. No Android file was changed.

This machine is Windows, not macOS, so Xcode itself is not available here. Per this issue's explicit scope, **Xcode validation is PENDING ON MACOS** — it has not been claimed as passing, and must be verified by opening `frontend/ios/App/App.xcodeproj` (or the workspace inside it) in Xcode on a Mac before any further iOS work (APNs, Firebase, native login, push, TestFlight, App Store, icons/splash customization) begins.

## Scope

**In scope (done):**
- Added `@capacitor/ios` as a dependency, matching the existing Capacitor 8.4.1 version used by `@capacitor/android`/`@capacitor/core`/`@capacitor/cli`.
- Generated the iOS native project (`frontend/ios/`) via `npx cap add ios`.
- Synced web assets and plugins via `npx cap sync ios`.
- Verified the generated project structure and confirmed `appId`/`appName`/`webDir` match the existing `capacitor.config.ts`.

**Explicitly out of scope (not done):** APNs, Firebase, native login, push notifications, TestFlight, App Store setup, iOS auth redirect changes, and iOS icon/splash customization beyond what Capacitor's `cap add ios` generates by default (its stock placeholder icon/splash assets, unmodified).

## What Changed

### 1. `frontend/package.json` / `frontend/package-lock.json`

Added `@capacitor/ios: ^8.4.1` as a dependency, matching the already-installed `@capacitor/android`, `@capacitor/core`, and `@capacitor/cli`, all at `8.4.1`. No version conflicts (`npm ls` confirms all four resolve cleanly with `@capacitor/core@8.4.1` deduped across all of them).

### 2. `frontend/eslint.config.js`

Added `'ios'` to the existing `globalIgnores(['dist', 'android'])` list, alongside the pre-existing `'android'` entry. This is required for the same reason `android` is already excluded: the generated platform directory contains a copy of the built web bundle (`ios/App/App/public/assets/*.js`), which is generated/minified output, not source code, and should not be linted. Without this, `npx eslint .` reported 167 errors, all inside the single copied bundle file. With it, lint is clean.

### 3. `frontend/ios/` (new, native iOS project)

Generated by `npx cap add ios`, then synced by `npx cap sync ios`. See "Generated Structure" below for the file list. No manual edits were made to any generated file.

## Known Risk: Windows-Generated Path Separator in `Package.swift`

`ios/App/CapApp-SPM/Package.swift` was generated on Windows, and one dependency path uses Windows-style backslashes rather than the POSIX forward slashes Swift Package Manager expects on macOS:

```swift
.package(name: "CapacitorPushNotifications", path: "..\..\..\node_modules\@capacitor\push-notifications")
```

This is a byproduct of generating the project on a non-macOS machine. **This must be checked when Xcode validation happens on macOS** — if Xcode/SPM fails to resolve this local package path, the fix is to re-run `npx cap sync ios` (or `npx cap update ios`) directly on the macOS machine that will do Xcode validation, which should regenerate this file with correct POSIX separators. This is flagged here specifically so the macOS validator checks for it rather than being surprised by a path-resolution error.
