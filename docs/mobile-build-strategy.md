# Mobile Build Strategy

**ISSUE:** 187
**Date:** 2026-06-30
**Status:** Approved strategy reference
**Scope:** Documentation and audit only - no builds generated, no signing keys created, no store releases

---

## 1. Purpose

This document defines the official build strategy for the Yesh Mishak mobile application. It establishes how Debug, Internal Testing, Beta, and Release builds are handled, including environment mapping, app identifiers, signing expectations, distribution channels, and release safety gates.

This strategy builds on:

- ISSUE-184: Mobile environment strategy (development, staging, production isolation)
- ISSUE-185: Mobile configuration management (VITE_API_URL canonical, no frontend secrets)
- ISSUE-186: Native plugin governance (plugin approval before release)

## 3. Debug Builds

### 3.1 Purpose

- Local development and fast iteration
- Emulator and device testing by developers
- Debugging with dev tools

### 3.2 Configuration

| Property | Value |
| :--- | :--- |
| Environment | Development |
| Android applicationId | `com.yeshmishak.app.dev` |
| iOS Bundle Identifier | `com.yeshmishak.app.dev` |
| Display name | יש משחק Dev |
| Backend | Local FastAPI (`http://localhost:8000`) or dev deployment |
| Supabase | Dev / local Supabase project |
| Firebase | Dev Firebase project |

### 3.3 Signing

- Android: debug keystore only (auto-generated by Android SDK)
- iOS: local developer signing only (when iOS project exists)

### 3.4 Distribution

- Local install only (ADB, Xcode direct install)
- Not for external testers
- Not for store upload

### 3.5 Rules

1. Debug builds must never connect to production services.
2. Debug builds must never be uploaded to Play Store or App Store.
3. Debug builds may include dev diagnostics only if guarded by `import.meta.env.DEV`.
4. Debug builds may use localhost / dev URLs only.

## 4. Internal Testing Builds

### 4.1 Purpose

- Internal QA validation
- Device compatibility testing across the team
- Pre-beta smoke testing

### 4.2 Configuration

| Property | Value |
| :--- | :--- |
| Environment | Staging |
| Android applicationId | `com.yeshmishak.app.staging` |
| iOS Bundle Identifier | `com.yeshmishak.app.staging` |
| Display name | יש משחק Staging |
| Backend | Staging backend deployment |
| Supabase | Staging Supabase project |
| Firebase | Staging Firebase project |

### 4.3 Signing

- Android: release-like signing may be used for internal distribution, but not production signing unless explicitly approved
- iOS: internal / TestFlight signing under staging bundle identifier (when iOS project and Apple setup exist)

### 4.4 Distribution

- Android: internal app sharing, internal testing track, or direct APK/AAB distribution
- iOS: TestFlight internal testing under staging bundle identifier (when available)

### 4.5 Rules

1. Internal testing builds must not use production backend, Supabase, or Firebase.
2. Internal testing builds must be clearly labeled as Staging.
3. Internal testers must not confuse this build with the production app.
4. Extra diagnostics are allowed only if not exposed in production builds.

## 5. Beta Builds

### 5.1 Purpose

- Wider tester validation before public release
- Closed beta / external beta testing
- Production-like UX validation with staging data

### 5.2 Default Configuration

| Property | Value |
| :--- | :--- |
| Environment | Staging |
| Android applicationId | `com.yeshmishak.app.staging` |
| iOS Bundle Identifier | `com.yeshmishak.app.staging` |
| Display name | יש משחק Staging or יש משחק Beta |
| Backend | Staging backend deployment |
| Supabase | Staging Supabase project |
| Firebase | Staging Firebase project |

### 5.3 Distribution

- Android: Google Play closed / internal testing track for staging app
- iOS: TestFlight external testing for staging app (when available)

### 5.4 Rules

1. Default beta builds use staging services.
2. Beta testers must not affect real production data unless explicitly approved.
3. Push notification tokens must remain separated by environment.
4. Beta builds must not silently point to production.

### 5.5 Release Candidate Exception

A Release Candidate is a beta build promoted to use production identity and services:

| Property | Release Candidate Value |
| :--- | :--- |
| Android applicationId | `com.yeshmishak.app` |
| iOS Bundle Identifier | `com.yeshmishak.app` |
| Backend | Production |
| Supabase | Production |
| Firebase | Production |

Release Candidate rules:

1. Requires explicit owner approval before using production identifiers or services.
2. Must pass the full release safety checklist (Section 9).
3. Must not include dev/test controls or debug diagnostics.
4. Is the final validation step before a Release build.

## 6. Release Builds

### 6.1 Purpose

- Real users with real data
- Public store distribution
- Production environment only

### 6.2 Configuration

| Property | Value |
| :--- | :--- |
| Environment | Production |
| Android applicationId | `com.yeshmishak.app` |
| iOS Bundle Identifier | `com.yeshmishak.app` |
| Display name | יש משחק |
| Backend | Production backend deployment |
| Supabase | Production Supabase project |
| Firebase | Production Firebase project |

### 6.3 Signing

- Android: production release keystore only (not debug keystore)
- iOS: production signing with App Store distribution provisioning profile

### 6.4 Distribution

- Android: Google Play production track
- iOS: Apple App Store production release

### 6.5 Rules

1. Release builds must never contain localhost, staging, dev, or test labels.
2. Release builds must not expose secrets.
3. Release builds must use production app identifiers only.
4. Release builds must pass all release safety checks (Section 9) before distribution.
5. Release builds must not include runtime environment switching.
6. Release builds must not include dev-only feature flags or test push controls.

## 7. Build-Time Configuration

### 7.1 Environment Selection

Build type determines environment. Environment is selected at build time, not at runtime.

| Build Type | Env File | VITE_APP_ENV (future) |
| :--- | :--- | :--- |
| Debug | `.env.development` | `development` |
| Internal Testing | `.env.staging` | `staging` |
| Beta | `.env.staging` | `staging` |
| Release | `.env.production` | `production` |

### 7.2 Configuration Rules

1. Build type must select environment at build time (per ISSUE-184, ISSUE-185).
2. Runtime user-facing environment switch is forbidden.
3. Canonical API variable is `VITE_API_URL`. `VITE_API_BASE_URL` is backward-compatible fallback only (per ISSUE-185).
4. Build must load the correct env file or CI variables for the target environment.
5. Production build must fail or be blocked if required production variables are missing.
6. No manual code edits to switch environments.

### 7.3 Future Build Scripts

These scripts are not yet implemented. They are documented as the recommended pattern for future implementation:

**Frontend builds:**

```
npm run build:development       # Debug environment
npm run build:staging           # Internal Testing / Beta
npm run build:production        # Release
```

**Capacitor sync:**

```
npm run cap:sync:development
npm run cap:sync:staging
npm run cap:sync:production
```

**Platform builds:**

```
npm run android:debug           # Debug APK
npm run android:internal        # Internal testing AAB
npm run android:beta            # Beta AAB
npm run android:release         # Release AAB
npm run ios:debug               # Debug IPA (future)
npm run ios:internal            # Internal TestFlight (future)
npm run ios:beta                # Beta TestFlight (future)
npm run ios:release             # Release IPA (future)
```

## 8. Build Artifacts

### 8.1 Naming Convention

Recommended artifact naming pattern:

```
yesh-mishak-{platform}-{buildType}-{environment}-{version}-{buildNumber}.{ext}
```

Examples:

| Artifact | Description |
| :--- | :--- |
| `yesh-mishak-android-debug-development-0.1.0-12.apk` | Debug APK for local testing |
| `yesh-mishak-android-internal-staging-0.1.0-18.aab` | Internal testing AAB |
| `yesh-mishak-ios-beta-staging-0.1.0-22.ipa` | Beta IPA for TestFlight |
| `yesh-mishak-android-release-production-1.0.0-100.aab` | Production release AAB |

### 8.2 Versioning Expectations

| Rule | Details |
| :--- | :--- |
| Android `versionCode` | Must be monotonically increasing for production releases. Single source of truth, current value, validation, and bump procedure are defined in `docs/e10-05-android-version-strategy.md`. Currently `1`. |
| Android `versionName` | Semantic version string (`MAJOR.MINOR.PATCH`), validated by Gradle and CI per `docs/e10-05-android-version-strategy.md`. Currently `"1.0.0"`. |
| iOS build number | Must be increasing for each TestFlight / App Store upload |
| Git traceability | Every distributed build must identify source commit SHA |
| Beta/internal builds | Must be traceable to git commit SHA |
| Release notes | Should include environment, build type, and commit reference |

### 8.3 Current Versioning State

From `frontend/android/app/build.gradle` (single source of truth, `appVersionCode`/`appVersionName`):

- `versionCode 1` - initial baseline; no artifact has ever been uploaded to Play Console (E10-05 audit), so this remains a safe starting point
- `versionName "1.0.0"` - normalized to strict `MAJOR.MINOR.PATCH` in E10-05 (previously `"1.0"`); both values are now Gradle- and CI-validated on every build (`docs/e10-05-android-version-strategy.md`)

## 10. What Must Never Happen

| Rule | Reason |
| :--- | :--- |
| Production build connected to staging services | Real users would see test data |
| Staging/debug build connected to production services | Test operations would corrupt real data |
| Manual code edits to switch environments | Error-prone, bypasses build-time selection |
| Runtime environment selector visible to users | Users could accidentally switch environments |
| Store upload from dirty working tree | Build is not reproducible |
| Store upload without known commit SHA | Build is not traceable |
| Secrets embedded in frontend/mobile bundle | Secrets would be visible to anyone who decompiles the app |
| Test push controls in production | Users could trigger unintended notifications |
| New plugin added during build/release without ISSUE-186 approval | Bypasses governance policy |

## 12. Summary

The mobile build strategy defines four build types with strict environment isolation:

1. **Debug** - development environment, local only, debug signing, never touches production
2. **Internal Testing** - staging environment, team QA, staging signing, internal distribution
3. **Beta** - staging by default, wider testers, Release Candidate exception for production after approval
4. **Release** - production only, production signing, public stores, full safety checklist required

Every build selects its environment at build time. No runtime switching. No manual code edits. No secrets in frontend. Production releases require a 21-point safety checklist before distribution.
